👤USER_ACCOUNTSUPERTYPE / AUTHENTICATION
FIX CRITICAL Removed polymorphic profile_id + profile_type anti-pattern. Replaced with four typed nullable 1:1 FK columns — one per subtype. Only one of these will be non-null for any given user row. This is the academically correct supertype/subtype relational mapping (Option B — FKs in the supertype table).
PKuser_idINT AUTO_INCREMENT
UKusernameVARCHAR(50) NOT NULL
UKemailVARCHAR(100) NOT NULL
password_hashVARCHAR(255) NOT NULL
roleENUM(doctor/patient/nurse/employee/admin)
FKNULLpatient_id→ PATIENT (nullable)
FKNULLdoctor_id→ DOCTOR (nullable)
FKNULLnurse_id→ NURSE (nullable)
FKNULLemployee_id→ EMPLOYEE (nullable)
Academic constraint: exactly one of (patient_id, doctor_id, nurse_id, employee_id) must be non-null; the rest null. Enforced via CHECK constraint or application logic. CHECK: (patient_id IS NOT NULL)::int + (doctor_id IS NOT NULL)::int + (nurse_id IS NOT NULL)::int + (employee_id IS NOT NULL)::int = 1 (or 0 for admin).
USER_ACCOUNT (PK user_id, UK username, UK email, password_hash, role, FKNULL patient_id, FKNULL doctor_id, FKNULL nurse_id, FKNULL employee_id)

🏛️DEPARTMENT_LOCATIONMULTIVALUED ATTRIBUTE
FIX CRITICAL Replaced surrogate PK dept_location_id with composite PK (department_code, location). Per Elmasri & Navathe, the correct relational mapping of a multivalued attribute is a new relation whose PK = {original entity PK + the attribute}. A surrogate PK here would lose this constraint and allow duplicate location values for the same department.
PKFKdepartment_code→ DEPARTMENT
PKlocationVARCHAR(255)
DEPARTMENT_LOCATION (PKFK department_code, PK location)
